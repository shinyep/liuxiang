# 代码质量审核报告 — 成品追溯与客户管理系统

> 审核日期：2026-05-23 | 审核范围：全项目（后端 Django + 前端 Vue 3）

---

## 严重问题（6 项）

### 1. 安全：硬编码的 SECRET_KEY

- **文件**: `flow_system/flow_system/settings.py`
- **行号**: 40
- **代码**:

```python
SECRET_KEY = 'django-insecure-8m-3rr^1dtbp-(m_1d!i))znkocxa-0r$w-#g9xm-by@2x+t44'
```

- **问题描述**: Django 的 SECRET_KEY 用于加密签名（session、CSRF token 等）。硬编码并暴露在源码中，任何拿到代码的人都可以伪造签名。
- **建议**: 改为从环境变量读取，如 `SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY', 'dev-key')`。

---

### 2. 安全：生产环境配置未关闭

- **文件**: `flow_system/flow_system/settings.py`
- **行号**: 43-45, 215
- **代码**:

```python
DEBUG = True
ALLOWED_HOSTS = ['*']
CORS_ALLOW_ALL_ORIGINS = True
```

- **问题描述**: 三项配置组合意味着：调试信息对外暴露、任何主机均可访问、任何来源可跨域请求。虽然是局域网部署，一旦网络边界被突破，攻击者可获取完整错误堆栈和内部路径。
- **建议**: 
  - `DEBUG` 通过环境变量控制
  - `ALLOWED_HOSTS` 限制为实际使用的 IP/域名
  - `CORS_ALLOW_ALL_ORIGINS` 按需限制

### 3. 功能缺陷：管理命令使用了已删除的数据库字段

- **文件**: `flow_system/api/management/commands/import_shipments.py`
- **行号**: 194-198
- **代码**:

```python
ProductionTimeSlot.objects.create(
    shipment=shipment,
    start_time_str=start_time_str,
    end_time_str=end_time_str,
    quantity=slot_quantity  # 此字段已删除
)
```

- **问题描述**: `ProductionTimeSlot` 模型的 `quantity` 字段已通过迁移 `0004_remove_productiontimeslot_quantity` 删除，但该管理命令仍尝试写入该字段。运行时将抛出 `TypeError: ProductionTimeSlot() got an unexpected keyword argument 'quantity'`。
- **建议**: 移除 `quantity=slot_quantity` 参数。

const response = await axios.get(`/api/v1/statistics/customer-daily/?${params}`);

### 5. 功能缺陷：前端追加时间段调用了不存在的后端端点

- **文件**: `flow_frontend/src/components/ShipmentTracking.vue`
- **行号**: 1415-1418
- **代码**:

```javascript
const resp = await axios.post(
  `${API_URL}/shipments/${appendedShipmentId}/append-slots/`,
  { production_time_slots: slots }
);
```

- **问题描述**: `/api/v1/shipments/{id}/append-slots/` 端点未在 `api/urls.py` 中注册。当提交重复记录触发 409 冲突后，前端显示的"追加时间段"按钮实际无法工作，用户点击后请求会返回 404。
- **建议**: 在后端 `ShipmentViewSet` 中添加 `append_slots` action 并在路由中注册，或修改前端逻辑复用现有的 create 接口。

### 6. 安全：数据库导入/导出接口无任何认证

- **文件**: `flow_system/api/views.py`
- **行号**: 649-690
- **问题描述**: `DatabaseBackupView` 和 `DatabaseImportView` 均使用 `permission_classes = [permissions.AllowAny]`。任何能访问网络的人都可以：
  - 下载完整 SQLite 数据库（含全部客户和流向数据）
  - 上传任意 SQLite 文件覆盖当前数据库
- **建议**: 至少为这两个接口添加基本认证，如 Django session 认证或简单的 token 验证。此外，导入操作应在覆盖前先备份现有数据库。

## 中等问题（4 项）

### 7. 异常信息泄露到 API 响应

- **文件**: `flow_system/api/views.py`
- **行号**: 188-189, 466-468, 645-646 等多处
- **代码**:

```python
except Exception as e:
    print(f"Error in ShipmentViewSet create: {str(e)}")
    return Response({"error": f"An error occurred: {str(e)}"}, ...)
```

- **问题描述**: 多处 `except Exception` 捕获块将原始异常消息直接返回给前端。攻击者可能通过构造特殊请求获取数据库结构、文件路径等敏感内部信息。
- **建议**: 对外只返回通用错误提示，详细异常记录到日志中。

---

### 8. 统计视图对空 shipment_date 的过滤行为有误

- **文件**: `flow_system/api/views.py`
- **行号**: 431-436
- **代码**:

```python
shipments_in_group = Shipment.objects.filter(
    shipment_date=s_date,  # s_date 可能为 None
    ...
)
```

- **问题描述**: `grouping_keys` 中包含 `shipment_date=None` 的分组。当 `shipment_date` 为 `None` 时，该过滤器匹配所有 `shipment_date IS NULL` 的记录，导致不同外箱日期的记录被错误合并到同一分组中。这与前端 `(shipment_date || outer_box_date)` 的显示逻辑不一致。
- **建议**: 对 `s_date is None` 的情况特殊处理，使用 `Q(shipment_date__isnull=True)` 并额外按 `outer_box_date` 分组。

---

### 9. 导入时规格值转换静默丢失小数精度

- **文件**: `flow_system/api/management/commands/import_shipments.py`
- **行号**: 150-151
- **代码**:

```python
slot_quantity = int(float(str(slot_quantity_val)))
```

- **问题描述**: 如果 Excel 中规格值为 `"12.7"`，`int(float("12.7"))` 结果为 `12`，小数部分被静默丢弃。这不一定是业务期望的行为（可能需要四舍五入或报告异常）。
- **建议**: 根据业务需求改用 `round(float(...))` 或在值非整数时记录警告。

---

### 10. 启动器端口清理使用 Windows 专用 Shell 命令

- **文件**: `flow_system/launcher_gui.py`
- **行号**: 462-481
- **代码**:

```python
subprocess.run(f'netstat -ano | findstr :{self.port}', shell=True, ...)
subprocess.run(f'taskkill /PID {pid} /F', shell=True, ...)
```

- **问题描述**: `netstat`、`findstr`、`taskkill` 是 Windows 专属命令。虽然该启动器本就是 Windows 桌面应用，但 `stop_server` 方法的 fallback 路径在非 Windows 环境下会静默失败。同时代码整体混用 Unix/PowerShell 风格（如 `requirements.txt` 中写的是 PowerShell 命令）。
- **建议**: 明确标注 Windows-only 约束；使用 `platform.system()` 做兼容处理或使用 `psutil` 库替代 shell 命令。

## 建议改进（6 项）

### 11. 模型 DateTimeField 的 null/blank 参数无意义

- **文件**: `flow_system/api/models.py`
- **行号**: 7-8, 22-23
- **代码**:

```python
created_at = models.DateTimeField(auto_now_add=True, null=True, blank=True)
updated_at = models.DateTimeField(auto_now=True, null=True, blank=True)
```

- **问题描述**: 对于 `auto_now_add` 和 `auto_now` 字段，Django 会自动管理值，它们永远不会为 null。设置 `null=True` 和注释"暂时允许 null"是误导性的。
- **建议**: 移除 `null=True, blank=True` 参数。

---

### 12. 全部 API 视图无认证机制

- **文件**: `flow_system/api/views.py`
- **涉及**: `CustomerViewSet`、`ShipmentViewSet`、`ShipmentImportView`、全部统计视图等
- **问题描述**: 所有 View 均使用 `permission_classes = [permissions.AllowAny]`。即使是内网应用，对数据清空（`clear_all_shipments`）、数据库覆盖导入等危险操作也应增加认证。
- **建议**: 至少为危险操作添加 `IsAuthenticated` 或 `IsAdminUser` 权限。

---

### 13. 前端轮询间隔过短

- **文件**: `flow_frontend/src/components/ShipmentTracking.vue`
- **行号**: 1669
- **代码**:

```javascript
this.pollingId = setInterval(this.fetchShipments, 15000);
```

- **问题描述**: 每 15 秒全量拉取列表数据。在数据量较大或多用户同时操作时，频繁的全量查询会造成不必要的数据库负载。
- **建议**: 考虑延长至 60 秒，或改为 WebSocket 推送、或在用户操作后手动刷新。

---

### 14. 客户数据硬编码在 Python 源码中

- **文件**: `flow_system/api/management/commands/import_customers.py`
- **行号**: 10-645
- **问题描述**: 约 640 行真实客户数据（含公司名称、地址）直接硬编码在 Python 文件中作为字符串常量。数据和代码耦合，且客户信息变更需要修改源码。
- **建议**: 将客户数据存储为独立的 CSV/Excel 文件，管理命令从文件读取。

### 15. 测试完全缺失

- **文件**: `flow_system/api/tests.py`
- **代码**:

```python
from django.test import TestCase
# Create your tests here.
```

- **问题描述**: 项目中无任何自动化测试。后端涉及复杂的数据导入解析、创建/更新去重逻辑、统计聚合计算，没有测试覆盖意味着每次改动都有回归风险。
- **建议**: 至少为核心业务逻辑编写测试：
  - `ShipmentViewSet.create()` 的创建/匹配/追加时间段逻辑
  - `ShipmentImportView` 的 Excel 解析和时间段验证
  - 统计视图的聚合计算
  - Serializer 的验证逻辑

## 其他注意事项

| 事项 | 位置 | 说明 |
|------|------|------|
| 区域码重复 | `import_customers.py:10-645` | 客户数据中存在同一区域码多次出现的记录，`get_or_create` 只保留首次匹配的地址，后续记录的地址修正会被跳过 |
| `Serializer.update()` 对空列表行为不明确 | `serializers.py:157-174` | 当 `slots_data` 为 `None`（字段未传入）时不处理，为 `[]` 时删除全部已有时间段。如果前端意外省略该字段，已有时间段会被保留而非清空，这在 PATCH 语义下有争议 |
| Launcher GUI 重复 `delete` | `launcher_gui.py:387-389` | 端口被占用时连续调用了两次 `self.port_entry.delete(0, 'end')`，第二行冗余 |
| Django migration 标记非标准 | `settings.py:109` | 使用了非标准配置项 `'MIGRATE': False`，这不是 Django 官方支持的 DATABASES 选项，可能不生效 |
| 前端 `return` 的 `data()` 含注释 | `ShipmentTracking.vue:373-374` | `data()` 中有注释 `/* ... data ... */` 是之前从模板生成时的残留 |

## 总结

| 级别 | 数量 | 关键项 |
|------|------|--------|
| 严重 | 6 | 安全配置、功能不可用、数据丢失风险 |
| 中等 | 4 | 逻辑错误、精度丢失、平台兼容 |
| 建议 | 6 | 代码质量、测试、维护性 |

**建议优先修复：**

1. `import_shipments.py` 使用已删除的 `quantity` 字段 → Excel 导入功能崩溃
2. 两个前端调用不存在的后端端点 → 追加时间段和客户详情功能不可用
3. 数据库导入/导出无访问控制 → 数据安全风险
4. 硬编码 SECRET_KEY + DEBUG=True → 安全风险
5. 补充核心业务逻辑的自动化测试 → 防止回归
